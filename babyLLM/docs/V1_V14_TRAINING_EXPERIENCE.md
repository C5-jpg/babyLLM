## 1. 项目背景与约束条件

| 项目 | 详情 |
|------|------|
| 竞赛 | NLPCC 2026 ChineseBabyLM Challenge |
| 数据集 | `babylm-zho-100M` — 约 1 亿中文字符（~2M 行，清洗后 ~82M tokens） |
| 核心约束 | 从零预训练，不可使用外部预训练模型 |
| Token 限制 | ≤100M Jieba tokens |
| 评测流水线 | [evaluation-pipeline-2025](https://github.com/SiyuanSong2004/evaluation-pipeline-2025) |
| 训练周期 | 2026-04-19 至 2026-05-12（~23 天，14 个版本） |

**核心矛盾**: 100M tokens 的数据量远低于 Chinchilla 最优比例（tokens/params = 20-200×），这意味着模型参数量必须严格控制在 ~5-50M 范围内才能充分训练。

## 4. 逐版本深度分析

### 4.1 V1 — GPT-2 基线（2026-04-19）

**架构**: GPT-2, 768d/12L/12H, ~110M 参数
**Tokenizer**: BPE 32K（空格+标点分词）
**训练**: 10 epochs, lr=6e-4, Cosine 调度器, fp32 精度

**关键发现**:
- LR 调度器 Bug: `num_training_steps` 计算错误，导致 LR 提前衰减至零
- **意外收益**: Bug 创造了周期性 LR 重启效果（类似 SGDR），PPL 从 490→343（-30%）
- 这个意外发现后来在 V11+ 中被有意实现为 SGDR 调度器

**教训**:
1. 代码 Bug 可能带来意外收益，但不应依赖 Bug
2. 周期性 LR 重启对小数据训练有显著帮助
3. fp32 精度在后续版本中被 bf16 替代（速度提升 ~2×）

### 4.2 V2 — LLaMA 架构迁移（2026-04-19）

**架构**: LLaMA, 768d/12L/12Q/4KV GQA, ~125M 参数
**Tokenizer**: ByteLevel BPE 32K
**训练**: 25 epochs, lr=6e-4, bf16, Flash Attention, BPE Dropout

**关键发现**:
- **PPL 反而恶化**: 343(V1) → 597(V2)，+74%
- 尽管 LLaMA 架构在每个组件上都优于 GPT-2（RoPE > Learned PE, SwiGLU > GELU, RMSNorm > LayerNorm, GQA > MHA）
- **根因**: ByteLevel BPE 将每个中文字符拆分为 3 个 UTF-8 字节，信息密度降低 3.4×
- Token/字符比: V1=0.541, V2=0.569（字节级）, V3=1.3（SPM）

**严重过拟合**:
- Train loss: 5.99→4.80（-20%）
- Val loss: 6.39→6.40（无改善）
- Gap: 0.40→1.60
- **浪费**: 18 epochs × ~25 min = ~7.5 小时（72% 训练时间）

**教训**:
1. **Tokenizer 选择 > 架构选择**: ByteLevel BPE 对中文的破坏性影响完全抵消了架构优势
2. **早期停止是必须的**: 100M 数据仅支持 ~6-10 个有效 epoch
3. 贪婪解码产生完全的 token 级重复崩溃

### 4.3 V3 — SentencePiece 修复（2026-04-20）

**架构**: LLaMA, 768d/12L, ~125M 参数
**Tokenizer**: SentencePiece BPE 32K
**训练**: WSD 调度器, 早期停止

**关键发现**:
- SPM 修复了 tokenizer 问题，PPL: 597→542（-9.2%）
- Token/字符比恢复到 ~1.3（正常水平）
- 但 WSD 调度器的稳定阶段鼓励了过拟合

**失败**: NCCL 超时（`ProcessGroupNCCL watchdog caught collective operation timeout`）
- 根因: 分布式训练同步问题，可能因 GPU 负载不均
- 修复: 设置 `NCCL_TIMEOUT=1800000`, `NCCL_IB_DISABLE=1`, `TORCH_NCCL_ASYNC_ERROR_HANDLING=1`

**教训**:
1. SentencePiece 是中文 LM 的正确 tokenizer 选择
2. WSD 调度器在小数据集上不适合（稳定阶段鼓励过拟合）
3. NCCL 环境变量配置对稳定性至关重要

### 4.4 V4 — 深层模型灾难（2026-04-20）

**架构**: LLaMA-deep, 1024d/24L/16Q/8KV, ~350M 参数
**Tokenizer**: SPM BPE 32K
**训练**: 50 epochs, lr=3e-4

**关键发现**:
- **完全失败**: 训练未收敛
- tokens/param = 0.23（理想值 20-200×）
- 模型欠训练 85×
- Chinchilla 违规: 350M 参数需要 ~7B tokens，实际只有 82M
- 额外问题: rope_theta=50000（对 1024 seq_len 过高），GQA 2:1（次优）
- SSD 空间也耗尽

**教训**:
1. **Chinchilla 法则验证**: 在不增加数据的情况下扩大参数是适得其反的
2. tokens/param 比率决定了训练天花板
3. rope_theta 应与 seq_len 匹配（10000 适合 1024）

### 4.5 V5 — 小模型 + 知识蒸馏（2026-04-21）

**架构**: LLaMA-small, 512d/12L/8Q/4KV, ~51M 参数
**Tokenizer**: SPM BPE 32K
**训练**: Phase 1（15 epochs CLM）+ Phase 2（KD 从 V2 教师）

**关键发现**:
- tokens/param = 1.61×（首次达到合理范围）
- PPL: 542→525（-3.1%）
- 更小的模型从相同数据中学到了更多

**失败**: SSD 空间耗尽 → 权重保存静默失败
- `save_pretrained()` 写入了 config JSON 但模型权重保存失败
- 训练日志显示 "Writing model shards: 100%" 但无 `model.safetensors` 文件
- 修复: 创建 symlink `output/` → `/mnt/sda/kehe/babyllm_output/`

**教训**:
1. **缩小模型可能比扩大模型更有效**（在数据有限时）
2. 必须验证权重文件确实存在且大小正确
3. 始终将输出目录 symlink 到大容量存储

### 4.6 V6 — 数据管线失败（2026-04-22）

**架构**: LLaMA, 640d/12L/10Q/5KV, ~75M 参数
**训练**: 3 阶段（CLM → CLM+MLM 1:7 MNTP → Reverse KL KD）

**关键发现**:
- 数据清洗过于激进: `min_length=15, max_length=300`
- 数据从 129.5M 字符 / 73.7M tokens 降至 28.2M 字符 / 15.5M tokens（-78%）
- tokens/param = 0.21 → 灾难性欠拟合

**额外 Bug**:
- MLM 标签移位错误（`mlm_labels[:-1] = input_ids[1:]`）
- 使用 `vocab_size - 1`（中文字符"講"）作为 mask token 而非 `<mask>`

**教训**:
1. **数据清洗必须保守**: 过度清洗比不清洗更糟糕
2. max_length 过滤应设为 1000+ 而非 300
3. MLM 实现需要仔细验证 mask token 和标签对齐

### 4.7 V7 — MNTP 混合训练（2026-04-23）

**架构**: LLaMA, 448d/12L/7Q/4KV, ~35M 参数
**Tokenizer**: SPM Unigram 8K（新训练，包含 `<mask>` token）
**训练**: CLM+MNTP 混合, 8K 词表

**关键发现**:
- 引入 MNTP（Masked Next Token Prediction）训练
- 8K 词表对中文来说太小
- PPL ~50.8

**教训**:
1. MNTP 混合训练是有效的（GPT-BERT 2024 冠军技术）
2. 8K 词表对中文不够，32K 更合适
3. 需要 `<mask>` token 支持 MNTP

### 4.8 V8-V9 — 探测与优化（2026-04-24/25）

**V8**: 简化 3 阶段管线, PPL=50.84
**V9**: 探测实验（MNTP stride、label smoothing、BPE dropout）

**关键发现**:
- stride=512（50% 重叠）比 stride=1024（无重叠）效果更好
- label_smoothing=0.02 有轻微帮助
- BPE dropout 可选但推荐

### 4.9 V10 — 生产级管线（2026-04-26）

**架构**: LLaMA, 512d/12L/8Q/4KV, ~38.7M 参数
**Tokenizer**: SPM Unigram 32K
**训练**: 3 阶段管线（CLM → MNTP → Polish）

| 阶段 | 最佳 Loss | 最佳 PPL | 步数 | 时间 |
|------|----------|---------|------|------|
| S1 CLM | 4.018 | 55.60 | 4,803 | 41 min |
| S2 MNTP | 3.926 | 50.68 | 12,808 | 89 min |
| S3 Polish | **3.759** | **42.89** | 2,826 | 20 min |

**关键创新**:
- 生产级代码质量
- WandB 监控集成
- 周期性检查点保存
- pin_memory + prefetch_factor=4 + persistent_workers

### 4.10 V11 — EMA + SGDR + 自蒸馏（2026-04-26）

**架构**: LLaMA, 512d/12L, ~38.7M 参数
**训练**: 5 阶段（CLM SGDR → MNTP → Polish → Annealing → Self-Distill）

| 阶段 | 最佳 Loss | 最佳 PPL | 步数 | 时间 |
|------|----------|---------|------|------|
| S1 CLM SGDR | 3.816 | 45.44 | 8,478 | 72 min |
| S2 MNTP | 3.710 | 40.85 | 11,304 | 81 min |
| S3 Polish | 3.708 | 40.78 | 7,065 | 50 min |
| S4 Annealing | 3.708 | 40.76 | 4,239 | 30 min |
| S5 Self-Distill | **3.707** | **40.73** | 5,652 | 53 min |

**关键创新**:
- **SGDR 调度器**: 周期性 LR 重启（T_mult=2），验证了 V1 的意外发现
- **EMA (decay=0.999)**: Stage 1 提供 8.4% PPL 改善
- **自蒸馏**: T=2.0, λ=0.5
- **SWA**: 尝试但效果微弱（0.02% 差异）
- **动态 CLM 比例**: 25%→12.5%→6.25%
- **Label Smoothing 退火**: 0.1→0

**教训**:
1. EMA 在 Stage 1（高 LR、高 dropout）效果最显著
2. SGDR 有效触发损失面重新探索
3. 自蒸馏提供边际改善
4. SWA 需要更长的平均周期

### 4.11 V12 — Focal Loss + 数据清洗（2026-04-27）

**架构**: LLaMA, 576d/14L/9Q/3KV, ~54M 参数
**训练**: 5 阶段 + Focal Loss

| 阶段 | 最佳 Loss | 最佳 PPL | 步数 | 时间 |
|------|----------|---------|------|------|
| S1 CLM SGDR | 3.718 | 41.19 | 11,296 | 114 min |
| S2 MNTP | 3.662 | 38.94 | 14,120 | 118 min |
| S3 Polish | 3.664 | 39.01 | 11,296 | 94 min |
| S4 Self-Distill | 3.665 | 39.04 | 8,472 | 90 min |
| S5 Annealing | 3.665 | 39.05 | 4,236 | 36 min |

**关键创新**:
- **Focal Loss (γ=2.0)**: 聚焦困难样本
- **更大模型**: 38.7M→54M, tokens/param=1.9×
- **数据清洗**: min_chars=5, max_repeat_ratio=0.5, MD5 去重
- **改进的自蒸馏**: T=4.0, λ=0.7, 每 100 步更新教师

**教训**:
1. Focal Loss 对 MNTP 训练有帮助
2. 54M 参数是 100M tokens 的合理上限
3. Stage 3+ 收益递减（S3 PPL 39.01 vs S2 38.94）

### 4.12 V13 — 最大模型（2026-04-28）— 当前 SOTA

**架构**: LLaMA, 768d/14L/12Q/4KV, ~94.2M 参数
**训练**: 3 阶段 + PPL 过滤数据

| 阶段 | 最佳 Loss | 最佳 PPL | EMA PPL | 步数 | 时间 |
|------|----------|---------|---------|------|------|
| S1 CLM SGDR | 3.742 | 42.19 | 39.51 | 22,432 | 188 min |
| S2 MNTP | 3.707 | 40.73 | **38.68** | 28,040 | 185 min |
| S3 Polish | 3.697 | 40.32 | 40.08 | 14,020 | 92 min |

**关键创新**:
- **PPL 过滤**: 使用 V12 模型过滤低质量数据（max_ppl=200）
- **MinHash 去重**: threshold=0.7, 128 permutations, 3-grams
- **硬样本上采样**: PPL>80 的样本 ×2
- **DropBlock 注意力**: p=0.1, block_size=3（仅 Stage 3）
- **随机深度**: p=0.05（仅 Stage 3）
- **课程 Focal Loss**: γ: 1.5→1.0→0

**关键失败**:
- **Stage 3 Polish 完全失败**: DropBlock + StochDepth 在低 LR 下过度正则化
- EMA PPL 从 Stage 2 的 38.68 恶化到 Stage 3 的 40.08
- **最终选择**: Stage 2 EMA（PPL=38.68）而非 Stage 3

**教训**:
1. **PPL 过滤数据是有效的**: 数据质量 > 数据数量
2. **Stage 3 Polish 是负面优化**: 在低 LR 下添加正则化是有害的
3. **tokens/param=1.1× 过低**: 94.2M 参数对 100M tokens 来说过大
4. EMA 贡献随训练阶段递减: S1=8.4%, S2=6.7%, S3=0.8%

### 4.13 V14 — 效率优化（2026-04-29 至 2026-05-12）

**架构**: LLaMA, 640d/12L/10Q/5KV, ~52M 参数
**训练**: 5 阶段

| 阶段 | 最佳 Loss | 最佳 PPL | 步数 | 时间 |
|------|----------|---------|------|------|
| S1 CLM SGDR | 3.822 | 45.68 | 13,000 | 78 min |
| S2 MNTP | 3.787 | 44.12 | 8,772 | 77 min |
| S3 Polish | 3.727 | 41.55 | 3,000 | 18 min |
| S4 Self-Distill | 3.722 | 41.35 | 8,316 | 59 min |
| S5 Annealing | 3.722 | 41.36 | 3,000 | 18 min |

**关键创新**:
- **效率优化**: tokens/param=1.9×（从 V13 的 1.1× 改善）
- **步级评估**: 每 500 步验证
- **OOM 恢复**: 自动降低 batch size
- **优雅关闭**: SIGTERM/SIGINT 信号处理
- **增强 WandB 监控**: GPU 显存跟踪

**教训**:
1. 52M 参数在 100M tokens 上训练更充分
2. 5 阶段管线收益递减（S4/S5 改善微弱）
3. 步级评估提供了更好的收敛监控

## 5. 关键技术演进

### 5.1 技术评估矩阵

| 技术 | 引入版本 | 效果 | 影响程度 | 最佳配置 |
|------|---------|------|---------|---------|
| **SentencePiece Tokenizer** | V3 | ✅ 高度有效 | ~74% PPL 影响 | SPM Unigram 32K |
| **EMA (decay=0.999)** | V11 | ✅ 高度有效 | 6-8% PPL（Stage 1） | decay=0.999 |
| **SGDR 调度器** | V1(bug), V11 | ✅ 有效 | 周期性重启 | T_mult=2, eta_min=1e-6 |
| **MNTP 混合训练** | V7 | ✅ 有效 | -2.21 PPL（Stage 2） | CLM:MNTP=1:7 |
| **Focal Loss** | V12 | ✅ 中等有效 | γ=2.0→1.5 | 聚焦困难样本 |
| **Label Smoothing 退火** | V11 | ✅ 有效 | 防止早期过度自信 | 0.1→0 |
| **BPE Dropout** | V2 | ✅ 有效 | 免费数据增强 | p=0.1 |
| **动态 CLM 比例** | V11 | ✅ 有效 | 课程学习 | 0.25→0.125→0.0625 |
| **PPL 数据过滤** | V13 | ✅ 有效 | 数据质量提升 | max_ppl=200-250 |
| **MinHash 去重** | V13 | ✅ 有效 | 去除近似重复 | threshold=0.7 |
| **自蒸馏** | V11 | ⚠️ 边际改善 | T=4.0, λ=0.7 | 收益有限 |
| **SWA** | V11 | ⚠️ 微弱 | 0.02% 差异 | 需要更长周期 |
| **DropBlock** | V13-S3 | ❌ 失败 | 过度正则化 | 低 LR 下有害 |
| **随机深度** | V13-S3 | ❌ 失败 | 过度正则化 | 低 LR 下有害 |
| **WSD 调度器** | V3 | ❌ 不适合 | 稳定阶段鼓励过拟合 | 小数据不适合 |
| **ByteLevel BPE** | V2 | ❌ 灾难性 | 3.4× 信息密度损失 | 中文禁用 |

### 5.2 EMA 贡献按阶段分析（V13）

| 阶段 | 基础 PPL | EMA PPL | 改善 | 原因 |
|------|---------|---------|------|------|
| Stage 1 (CLM) | 43.15 | 39.51 | **-8.4%** | 高 LR + 高 dropout = 噪声梯度 |
| Stage 2 (MNTP) | 41.45 | 38.68 | **-6.7%** | 中等 LR |
| Stage 3 (Polish) | 40.40 | 40.08 | -0.8% | 低 LR，收益递减 |

**结论**: EMA 在训练早期（高 LR、高正则化）价值最大。

### 5.3 调度器对比

| 调度器 | 版本 | 效果 | 适用场景 |
|--------|------|------|---------|
| Cosine | V1-V10 | 标准，稳定 | 通用 |
| WSD | V3 | 不适合 | 大数据集 |
| SGDR | V1(bug), V11+ | 有效 | 小数据集，需要周期性探索 |

## 6. 失败模式与根因分析

### 6.1 失败分类

| 类别 | 失败 | 版本 | 影响 |
|------|------|------|------|
| **Tokenizer** | ByteLevel BPE 对中文的破坏 | V2 | PPL +74% |
| **规模** | 350M 参数 / 82M tokens | V4 | 完全不收敛 |
| **数据** | 过度清洗丢失 78% 数据 | V6 | 灾难性欠拟合 |
| **存储** | SSD 满导致权重丢失 | V5 | 最佳模型丢失 |
| **训练** | 无早期停止导致过拟合 | V2 | 72% 时间浪费 |
| **分布式** | NCCL 超时 | V3 | 训练中断 |
| **正则化** | DropBlock+StochDepth 过度 | V13-S3 | PPL 回退 |
| **MLM** | mask token 和标签错误 | V6 | MLM 信号损坏 |
| **评估** | ZhoBLiMP 3 维度低于随机 | V13 | 学习了错误模式 |

### 6.2 根因分析总结

1. **Tokenizer 选择是最高优先级**: 一个错误的 tokenizer 选择（ByteLevel BPE）导致 PPL 恶化 74%，完全抵消了架构升级的所有收益。

2. **数据量是绝对瓶颈**: 所有版本都受制于 100M tokens。tokens/param 比率决定了训练天花板：
   - 0.23× (V4) → 完全失败
   - 0.66× (V2/V3) → 严重欠训练
   - 1.1× (V13) → 边际收益
   - 1.6-2.6× (V5/V10/V11) → 最佳范围

3. **过拟合是普遍问题**: 所有版本在 epoch 6-10 后过拟合。100M 数据仅支持 ~6-10 个有效 epoch。

4. **Stage 3 Polish 是负面优化**: V13 证明在低 LR 下添加 DropBlock/StochDepth 是有害的。

### 9.1 GPT-2 → LLaMA 迁移

| 组件 | GPT-2 (V1) | LLaMA (V2+) | 收益 |
|------|-----------|-------------|------|

| V10/V11 | 38.7M | 2.6× | 最佳效率 |
| V14 | 52M | 1.9× | 良好 |

## 12. 核心定律与经验法则

### 12.1 已验证的基本定律

1. **数据是绝对瓶颈**: 所有版本受制于 100M tokens
2. **Tokenizer > 架构**: Tokenizer 选择有 ~74% PPL 影响，架构升级影响被掩盖
3. **质量 > 数量 > 容量**: 数据清洗（+35% 质量）> 增加参数（+280%）
4. **Tokens/Param 比率决定训练天花板**: V4(0.23) 失败，V5(1.61) 成功
5. **所有版本在 epoch 6-10 后过拟合**: 100M 数据支持 ~6-10 个有效 epoch
6. **LR 重启是强大的**: V1 的 Bug 证明了 30% PPL 改善

### 12.2 优先级建议

**优先级 1: 缩小模型 + 提升数据效率**
- d_model 768→576-640, n_layer 14→12, 目标 ~45-55M 参数
- tokens/param 从 1.1× 提升到 ~2.0×

**优先级 2: 数据增强**
- 语法模式上采样 1.5×
- PPL 过滤阈值放宽（max_ppl=200→300-500）
- 领域平衡（对话/文学/学术/新闻）

**优先级 3: 移除 Stage 3 Polish**
- V13 证明 Stage 3 是负面优化
- 使用 2 阶段管线: CLM→MNTP

**优先级 4: 超参数调优**
- LR 网格搜索: 3e-4 到 1e-3
- Batch size: 增加到 24×2=48 或 32×2=64
- SGDR t_mult: 尝试 1 或 1.5

**优先级 5: 架构改进**
- RoPE base 10000→500000
- 词表优化: 32K→16K
- 逐层学习率

### 12.3 工程最佳实践

1. **始终 symlink 输出到 HDD**: 防止 SSD 空间耗尽
2. **验证权重文件**: 检查 `model.safetensors` 存在且大小正确
3. **监控磁盘空间**: 训练前检查 `shutil.disk_usage()`
4. **NCCL 配置**: timeout=30min, 禁用 IB, 启用异步错误处理
5. **早期停止是必须的**: patience=3-8（100M 数据）
6. **WandB 监控**: 对调试训练动态至关重要
